# CLAUDE.md — правила для агента (читается при каждом запуске)

> Этот файл короткий намеренно: он грузится в контекст при **каждом** запросе. Детали — в `DOCUMENTATION.md`.

## Проект
Block Blast Core + редактор уровней. Одна страница — `/editor`. Стек: Next.js 16 (App Router), React 19, TypeScript, Pixi.js 8 (игра), Zustand 5 (стейт), SCSS modules, Howler (звук — заглушка под реальные файлы, пока не играет). Эффекты всегда включены (тумблера нет).

## 📖 Перед работой — прочитай DOCUMENTATION.md
**[DOCUMENTATION.md](DOCUMENTATION.md)** — это источник правды по фактической реализации: карта файлов, типы, сигнатуры функций, поток данных Pixi↔React, шпаргалка «где что менять» (§10) и расхождения с ТЗ (§11).
Читай его **вместо** повторного сканирования кода. Если ответ есть в нём — не открывай исходники.

`block_blast_core_game_prompts.md` и `AGENTS.md` — это исходное ТЗ/промпты (что задумано), а не текущее состояние. При конфликте кода и ТЗ верь коду + DOCUMENTATION.md.

## 🔄 ГЛАВНОЕ ПРАВИЛО: код и документация меняются вместе
Любая правка кода, которая меняет что-либо из перечисленного, **обязана** сопровождаться правкой `DOCUMENTATION.md` в том же изменении:
- добавлена/удалена/переименована функция, тип, поле `LevelConfig`, экшен стора;
- изменена сигнатура или поведение функции из таблицы §4 / §10;
- добавлен/удалён файл, слой Pixi, секция редактора;
- изменён стек, команды, контент (`defaultLevels`, `figureShapes`);
- появилось новое расхождение с ТЗ → добавь пункт в §11.

Не завершай задачу, не сверившись: «затронул ли я что-то, описанное в DOCUMENTATION.md?». Если да — обнови соответствующий раздел. Чисто косметические правки (форматирование, опечатки в комментариях) документацию не требуют.

## Конвенции
- Архитектура FSD-подобная: `entities/game` (чистая логика, без React/Pixi) → `widgets/game-core` (Pixi+стор) и `widgets/level-editor` (React UI) → `shared`.
- Чистую игровую логику держать в `entities/game/lib` без побочных эффектов. Мутации состояния — в `gameStore.ts` и `GameScene.ts`.
- Ссылки на код — кликабельными markdown-ссылками, не бэктиками.
- После изменений проверяй сборку: `npm run build` (и `npm run lint`).
